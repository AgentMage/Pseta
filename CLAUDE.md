## What Pseta is

A coincidence-driven groove agent and open measurement framework for synchrony. It listens to a musician playing a drum pad and responds by maximizing ζ (coincidence density) — not by predicting what comes next. The Rust binary is a dumb MIDI pipe; all musical intelligence lives in Python.

The math is grounded in Anna Taranova's Ψ-model (PCT/IB2025/055633). The three operationalized quantities are:
- **ζ (density)** — rate of cross-stream coincidences per unit time. The primary metric everything optimizes.
- **σ (symmetry)** — structural mirroring between streams. Coherence multiplier.
- **ρ (resonance)** — amplification when 3+ streams coincide simultaneously.

## Current scope

**Proof of concept: two-stream synchrony measurement.**

A MIDI I/O system that plays a groove from the dataset, captures live drumming alongside it, computes ζ(t) and σ(t) across the two streams in real time, and displays synchrony data in the terminal. This validates the measurement framework directly before building any agent behavior.

**The two streams:**
- Stream 1 — MIDI playback from `datasets/groove-v1.0.0-midionly/` (known, fixed timing)
- Stream 2 — Live drum pad input captured via the Rust binary (nanosecond timestamps)

**Build targets:**
1. Rust binary: MIDI capture (live pads) + MIDI playback (dataset groove), simultaneously
2. `zeta.py`: ζ(t) and σ(t) computed across the two streams using the Φ kernel
3. pygame UI (`main.py`): timeline (groove + user + ζ(t) lanes), dual 4×4 pad grids, MIDI monitors, transport, port selection, stream pair configuration via pad color assignment
4. Session logger: every tick to JSONL, including both stream states

**The one validation question:** Does ζ rise measurably above permutation baseline when the drummer locks in with the groove, and drop when they don't?

**Explicitly out of scope right now:**
- Action scoring (LISTEN / MIRROR / COMPLEMENT) — comes after measurement is validated
- MIDI output generated by the agent — not yet
- Any UI beyond terminal output
- Biometric streams, EEG, or any multi-modal hardware

---

**Stages 2 and 3 are distant future development — not near-term roadmap.**

The broader build arc (biometrics → neural signals) described in the README represents a multi-year research direction contingent on Stage 1 producing valid results. Do not design current code around those stages. Build for two MIDI streams. Extensibility to additional streams is a structural property to preserve, not an active target.

## Architecture

Two processes, one pipe:

```
Rust (midi_capture) → JSONL stream → Python (groove_agent) → MIDI output
                                                ↓
                                    sessions/YYYY-MM-DD_HH-MM-SS.jsonl
```

**Rust** (`src/main.rs`): Real-time MIDI I/O with nanosecond wall-clock timestamps. Pure capture and output — no musical logic.

**Python** (`groove_agent.py`): The main loop — `observe → update state → score actions → act`. Reads JSONL from stdin (Rust pipe), computes ζ, scores the three actions via softmax, emits MIDI.

**Module responsibilities:**
- `state.py` — State dataclass + update functions (onset history, bpm_estimate, bpm_variance, beat_phase, bar_phase, groove_map, recurrence, ζ_current)
- `zeta.py` — ζ, σ, ρ computation (see implementation spec below)
- `actions.py` — Action scoring + MIDI execution. The three actions: `Action.LISTEN`, `Action.MIRROR`, `Action.COMPLEMENT`
- `logger.py` — Session JSONL logger
- `config.toml` — All tunable parameters (MIDI ports, tempo range, subdivision, ζ window, action scoring weights, σ recognition radius, θ_on, θ_off, α, β, γ, δ)

## `zeta.py` — concrete implementation spec

From `documents/Psi Model Final 26.11.pdf` (authoritative):

**Coincidence function Φ** (the core kernel):
```
Φ(x, y) = exp(−|x − y|² / 2σ²)
```
σ is the "recognition radius" — the most important tunable parameter in `config.toml`. Controls how close two stream values must be to register as a coincidence. Calibrate via cross-validation or entropy criterion.

**Coincidence density** (pairwise):
```
ζ(t) = Σ_{1 ≤ i < j ≤ N} Φ(Sᵢ(t), Sⱼ(t))
```
Normalized to [0,1]: `ζ(t) = (1/n) Σ X_{ij}(t)` where n = N(N−1)/2.

**Coincidence derivative** (the Ψ signal itself):
```
Ψ(t) = d/dt ζ(t) = Σ_{i<j} [∇₁Φ(Sᵢ,Sⱼ)·Ṡᵢ + ∇₂Φ(Sᵢ,Sⱼ)·Ṡⱼ]
```

**Intersection operator — binary first, then weighted**: The ∩ operator is binary at core: X_{ij}(t) ∈ {0,1}. A pair coincides if per-dimension distances D_k are below per-dimension thresholds θ_k. Φ then weights the aggregate. Do not collapse these into a single continuous similarity score — the binary layer prevents micro-fluctuations from corrupting ζ.

**Symmetry vector** σ is five-dimensional per stream pair:
```
V_{ij}(t) = [Δφ, Δf, Δτ, Δs, Δm]   # phase, frequency, timing, structure, motif
```

**Compute ζ₂, ζ₄, and σ** — not just ζ₂. Rhythm locks in quartets; ζ₄ captures amplification events that pairwise metrics miss.

**Computational complexity**: O(N²) in stream count. Use windowed sampling and sparse matrices when N grows beyond a few streams.

## `actions.py` — composite signal and activation

Action scoring uses the weighted composite:
```
S_score(t) = G(t) · [α·ζ₂ + β·σ₂ + γ·ζ₄ + δ·σ₄]
```
Weights (α, β, γ, δ) in `config.toml`. G(t) is a gate encoding musical context (bar position, confidence level).

**Activation R(t) requires two thresholds** (hysteresis):
```
R(t) = 1          if ζ(t) ≥ θ_on
R(t) = 0          if ζ(t) ≤ θ_off
R(t) = R(t−Δt)   otherwise
```
θ_on > θ_off. Prevents flicker at the boundary. Continuous alternative: `R(t) = 1 / (1 + exp(−k(ζ(t) − ζ_c)))`.

## Statistical validity

The CLT result in the paper establishes ζ → N(μ, σ²) as N→∞, enabling confidence intervals. Always run phase-shuffled or time-jittered surrogate streams to establish chance ζ levels. Without surrogate baselines, ζ > 0 has no statistical meaning.

## Session log schema

Every tick produces one JSONL line with: `t` (nanosecond timestamp), `event_type`, `state` (bpm_estimate, bpm_variance, density_map_confidence, recurrence, zeta_current), `action_scores` (listen/mirror/complement), `action_taken`, `zeta_before`, `zeta_after`.

Session logs are the training data foundation for all future ML work — log everything, every tick.

## Action scoring behavior

The three action scores are computed as expected ζ gain. Their relative ordering shifts as knowledge accumulates:
- `listen` wins early (high uncertainty → information value is high)
- `mirror` wins as bpm stabilizes and confidence rises
- `complement` wins as recurrence fills in (groove_map has enough data to find gaps)

This emergent behavioral arc is the validation target — not a programmed state machine.

## Stage 1 success criteria

- `bpm_variance` < 0.05 after 4–8 bars
- `density_map_confidence` > 0.5 after 4–8 bars
- Mirror notes landing within 30ms of player hits
- Action distribution visibly shifts listen → mirror → complement over session
- ζ above zero and trending upward in session log

## Conduct standards

### Epistemic category tagging

Every claim in code comments and documentation must be tagged:

- `[A]` — **Reproducible:** defined inputs, formula applied, measurable output, baseline included. The only category that counts as evidence.
- `[B]` — **Observational/Interpretive:** pattern observations, analogies, conceptual parallels. Never used as proof.
- `[C]` — **Planned/In progress:** not yet implemented. Must be marked as future work.

```python
# [A] ζ(t) computed from MIDI stream with permutation baseline — reproducible result
# [B] This peak corresponds to what musicians call "locking in" — interpretive only
# [C] TODO: extend to EEG stream — not yet implemented
```

Never present a `[B]` observation as `[A]` evidence.

### Baseline requirement

Every ζ(t) computation on real data must be accompanied by a permutation baseline. No result is reportable without it:

1. Compute ζ(t) on real streams → `zeta_real`
2. Shuffle at least one stream (preserving length and marginal distribution)
3. Compute ζ(t) on shuffled streams → `zeta_baseline`
4. Report mean, std, and whether real peaks exceed baseline by a statistically meaningful margin

If writing a ζ computation without a baseline yet, add:
`# WARNING: No baseline — result is not yet scientifically reportable`

### Falsifiability docstring

Every experimental function must state what result would indicate the model is wrong:

```python
def compute_zeta(streams, sigma, epsilon):
    """
    Compute coincidence density ζ(t) across input streams.

    Parameters:
        streams: list of 1D time series, same length, same sample rate
        sigma: Gaussian kernel width (recognition radius)
        epsilon: temporal proximity threshold in samples

    Returns:
        zeta: 1D array of coincidence density over time

    Falsification condition:
        If ζ(t) on real data does not exceed permutation baseline by at least
        one standard deviation at known groove-lock events, the coincidence
        detection approach requires revision.
    """
```

### Ethical constraints

- **(E1)** Do not implement features that could enable non-consensual monitoring, behavioral steering, or manipulation. Flag and refuse if a request approaches this.
- **(E2)** Code handling biometric or physiological data must include explicit consent documentation, data minimization, and opt-out capability. No persistent storage of identifiable data without justification.
- **(E3)** No hardcoded credentials, paths, or machine-specific assumptions. All dependencies pinned. Failures and negative results documented, not deleted.

### When in doubt

If uncertain whether a result is `[A]` or `[B]`, treat it as `[B]`. If uncertain whether a feature is ethical, flag it before implementing. If a result is surprising, document it accurately and design a falsifying test — do not explain it away.

The goal is not to confirm the framework. The goal is to find out if it's true.
